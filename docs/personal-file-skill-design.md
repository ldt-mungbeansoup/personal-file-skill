# personal-file Skill 设计文档

## 1. 背景与目标

`personal-file` 的目标是把 Obsidian Vault 变成一个长期可用的个人系统：它能帮助用户整理知识库、形成个人档案、记录身体/情绪/时间项目/财务数据，并支持学习闭环、复盘、状态诊断和决策辅助。

它不是一个全自动黑箱，也不是简单的笔记分类脚本。它的核心原则是：

- 本地 Markdown 优先，用户能手动阅读和维护。
- 原始记录追加优先，规范档案谨慎改写。
- 高风险操作先预览、再确认、可回滚。
- Skill 负责整理、连接、复盘和提示，用户保留最终主权。

首版不做：

- Obsidian 插件 UI。
- 强制打卡提醒。
- 大而全的生活流水账。
- 医学诊断、投资建议或其他高风险专业结论。
- 自动远程推送。
- 未经确认的删除、合并、覆盖。

## 2. 总体架构

系统分为三层：

1. Codex Skill：识别用户自然语言意图，决定读取范围、写入文件、确认策略和回滚策略。
2. Obsidian Vault：本地 Markdown 文件系统，是唯一数据源。
3. 本地 Git：用于检查点、审计和回滚。

推荐结构：

```text
Codex Skill
  ├─ 读取 Vault 目录和系统说明
  ├─ 根据意图路由到模块
  ├─ 生成变更计划
  ├─ 执行低风险追加
  └─ 对中高风险操作先预览

Obsidian Vault
  ├─ 原始记录
  ├─ 规范档案
  ├─ 派生视图
  └─ 系统模板与变更日志

本地 Git
  ├─ 批量整理前检查点
  ├─ 变更摘要
  └─ 按文件回滚
```

## 3. Vault 目录结构

顶层目录：

```text
00-首页与导航
10-个人档案
20-学习与项目
30-数据记录
40-知识库
90-系统
```

数据记录分为四大类：

```text
30-数据记录
├── 身体
│   ├── 身体指标
│   ├── 训练
│   ├── 饮食
│   └── 睡眠与恢复
├── 情绪
│   ├── 心情记录
│   ├── 精力与压力
│   └── 触发因素与应对
├── 时间与项目
│   ├── 时间投入
│   ├── 项目进展
│   └── 阶段复盘
└── 财务
    ├── 收入与支出
    ├── 资产与负债
    └── 预算与财务目标
```

个人档案中的“健康与身体状态”和“财务状态”只保存当前状态、目标和摘要；原始时间序列保存在 `30-数据记录`。

## 4. 模块职责

### 4.1 个人档案

负责维护动态自我模型和事实档案：

- 基本信息。
- 当前目标。
- 经历、项目与关键成果。
- 技能、能力与证据。
- 兴趣、偏好与价值观。
- 决策记录。
- 外部反馈。
- 健康与身体状态。
- 财务状态。

规则：

- 更新规范档案前必须展示差异。
- 重要变化同时追加关键事件或决策记录。
- 档案结论必须尽量链接到证据，不复制长文本。

### 4.2 学习辅助

负责形成学习闭环：

```text
学习目标 → 实践项目 → 问题清单 → 知识沉淀 → 练习测验 → 阶段复盘 → 调整计划
```

支持：

- 创建学习目标。
- 拆解学习项目。
- 记录学习问题。
- 生成练习和测验。
- 把问题沉淀成知识笔记。
- 根据复盘调整下一步计划。

### 4.3 数据记录与分析

负责身体、情绪、时间项目、财务四类数据的记录、校验和复盘。

规则：

- 追加型记录可直接写。
- 涉及个人档案状态更新时先预览。
- 跨模块分析必须列出证据来源。
- 健康和财务只做记录与观察，不做高风险专业结论。

### 4.4 知识库整理

负责收件箱处理和长期笔记治理：

- 补 Properties。
- 补标签。
- 补双链。
- 分类为概念、问题、资料、方法等笔记。
- 发现重复内容并提出合并建议。
- 生成索引页。

规则：

- 补元数据和双链属于低风险。
- 移动、重命名属于中风险，先预览。
- 合并、删除属于高风险，必须单独确认并建立 Git 检查点。

### 4.5 状态诊断

读取近期身体、情绪、时间项目和当前目标，输出可能关联、证据来源和待观察变量。

边界：

- 不做医学诊断。
- 不把相关性说成因果。
- 结论分为“有证据支持 / 猜测 / 需要继续观察”。

### 4.6 决策辅助

用于重要选择前整理目标、证据、利弊、机会成本、风险和验证点。

确认后可写入：

```text
10-个人档案/决策记录/
```

### 4.7 复盘转行动

把周/月/阶段复盘压缩成 1-3 个最小行动。每个行动必须：

- 可执行。
- 可验证。
- 有证据来源。
- 不依赖完美状态。

## 5. 数据格式与跨模块契约

四类基础模板：

1. 规范档案：描述当前状态，附证据和变更记录。
2. 追加事件：记录发生什么、为何重要、后续行动。
3. 结构化数据：Properties 标记领域和日期，正文用可读表格与说明。
4. 知识笔记：问题、当前结论、证据来源、适用边界。

统一规则：

- Properties 用于筛选和索引，正文承载完整语义。
- 原始记录追加优先；纠错保留修订说明。
- 规范档案是当前事实源，改写前展示差异。
- 派生视图必须列出来源，可以重建。
- 跨模块用双链，不复制长文本。
- Skill 不认识的人工字段默认保留，不擅自清理。

## 6. 权限与安全

风险分级：

| 风险 | 操作 | 策略 |
| --- | --- | --- |
| 低风险 | 新增记录、补标签、补双链、生成草稿 | 可直接执行 |
| 中风险 | 移动、重命名、批量分类、更新项目状态 | 先预览再执行 |
| 高风险 | 删除、合并、覆盖、敏感推断性改写 | 单独确认并建立检查点 |

安全边界：

- 不读取或输出 `.env`、API Key、Token。
- 不主动联网。
- 不自动推送远程仓库。
- 只读取完成任务所需的目录。
- 涉及健康、财务、心理等内容时保持观察性表达。

## 7. Git 检查点与回滚

策略：

- 低风险追加可不单独建检查点。
- 中高风险批量操作前创建本地 Git 检查点。
- 删除或合并前保留来源文件或快照。
- 回滚优先按文件回滚，不做整库重置。
- 回滚后生成恢复记录。

变更日志建议放在：

```text
90-系统/变更日志/
```

## 8. 首页导航与人工可读性

`00-首页与导航/首页.md` 是个人驾驶舱，包含：

- 当前状态。
- 快速记录入口。
- 正在推进。
- 近期复盘。
- 系统说明。

每个一级目录保留 README：

- 说明该目录放什么、不放什么。
- 列出常用模板。
- 解释字段含义。
- 标明助手能自动做什么、哪些需要确认。

命名规则：

- 事件记录：`YYYY-MM-DD-主题.md`
- 周复盘：`YYYY-Wxx-周复盘.md`
- 月复盘：`YYYY-MM-月复盘.md`
- 规范档案：固定中文名，如 `当前目标.md`
- 知识笔记：优先用问题或概念命名。

## 9. 模板体系与首批文件

模板分三层：

- 记录模板：追加式，记录一次事件或输入。
- 档案模板：当前状态源。
- 复盘模板：周/月/阶段视角。
- 知识模板：概念、问题、资料、方法。

首批核心文件：

```text
00-首页与导航/首页.md
00-首页与导航/快速记录入口.md
00-首页与导航/当前状态总览.md
00-首页与导航/近期复盘索引.md

10-个人档案/基本信息.md
10-个人档案/当前目标.md
10-个人档案/能力地图.md
10-个人档案/兴趣偏好与价值观.md
10-个人档案/健康与身体状态.md
10-个人档案/财务状态.md
10-个人档案/外部反馈.md
10-个人档案/决策记录/

20-学习与项目/行动实验/

90-系统/模板/
90-系统/字段说明.md
90-系统/助手使用说明.md
90-系统/变更日志/
```

## 10. Skill 使用方式

用户使用自然语言即可：

- `记录今天训练……`
- `更新我的当前目标……`
- `帮我建一个学习项目……`
- `整理收件箱……`
- `复盘这周……`
- `看看最近效率低和睡眠训练有没有关系……`
- `帮我做一个决策记录……`
- `把复盘变成下周三个行动……`

也支持短命令别名：

- `记录：`
- `复盘：`
- `整理：`
- `更新档案：`
- `做决策：`

## 11. 意图路由规则

内部固定意图：

| 意图 | 用途 | 默认策略 |
| --- | --- | --- |
| `record.body` | 身体指标、训练、饮食、睡眠恢复 | 追加直接写 |
| `record.mood` | 心情、压力、精力、触发因素 | 追加直接写，结论只标观察 |
| `record.time_project` | 时间投入、项目进展、阶段情况 | 追加直接写，改状态预览 |
| `record.finance` | 收支、资产负债、预算目标 | 原始记录直接写，财务状态需确认 |
| `profile.update` | 更新个人档案 | 必须差异预览 |
| `learning.loop` | 学习目标、计划、练习、复盘 | 新建可直接，改既有计划预览 |
| `knowledge.organize` | 收件箱、标签、双链、合并 | 元数据可直接，移动合并删除需确认 |
| `review.pattern` | 周/月/阶段复盘、找模式 | 草稿可直接，档案结论需确认 |
| `state.diagnose` | 状态诊断 | 只输出可能关联 |
| `decision.support` | 决策辅助 | 草稿可直接，落档需确认 |
| `review.to_action` | 复盘转行动 | 新建行动实验可直接，改目标需确认 |

歧义处理：

- 优先保留原始事实，不急着生成档案结论。
- 能拆成多条记录时拆分。
- 需要改规范档案时先展示建议更新。
- 不确定时只问一个短问题。

## 12. 实施顺序与验收标准

实施顺序保持 `B → D → C → A`：

| 阶段 | 模块 | 验收标准 |
| --- | --- | --- |
| B | 个人档案 | 核心档案、当前目标、能力地图、决策记录、外部反馈、健康/财务状态可创建 |
| D | 学习辅助 | 学习目标、项目结构、问题清单、练习测验、阶段复盘可创建 |
| C | 数据记录与分析 | 身体、情绪、时间项目、财务模板和复盘视图可创建 |
| A | 知识库整理 | 收件箱处理规则、元数据补全、双链、移动/合并/删除确认流程明确 |

验收方式：

- 使用初始化脚本在空目录生成 Vault 骨架。
- 检查所有一级目录、核心文件、模板是否存在。
- 人工阅读首页和 README，确认无需 AI 也能理解结构。
- 用 5 个典型输入模拟路由，确认风险策略正确。
- 确认不会读取 `.env`、API Key、Token，也不会主动联网。
